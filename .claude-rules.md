# NextCommerce Campaign Cart SDK - Claude Rules

## Project Overview

NextCommerce Campaign Cart SDK is a **TypeScript-based e-commerce SDK** that provides advanced cart management, checkout flows, and upsell capabilities for campaign pages. It uses a **data-attribute-driven architecture** where HTML elements are enhanced with JavaScript functionality based on `data-next-*` attributes.

**Key Technologies:**
- TypeScript 5.3+
- Zustand (state management)
- Vite (build tool)
- Data-attribute-driven progressive enhancement

---

## Architecture Overview

### 1. How It Works

The SDK follows a **progressive enhancement pattern**:

1. **Initialization** ([src/enhancers/core/SDKInitializer.ts](src/enhancers/core/SDKInitializer.ts))
   - Loads configuration from meta tags or `window.nextConfig`
   - Fetches campaign data from API
   - Initializes stores (cart, campaign, checkout, etc.)
   - Scans DOM for `data-next-*` attributes
   - Creates enhancer instances for each element

2. **DOM Scanning** ([src/enhancers/core/AttributeScanner.ts](src/enhancers/core/AttributeScanner.ts))
   - Uses MutationObserver to watch for new elements
   - Parses data attributes to determine which enhancers to apply
   - Creates and initializes appropriate enhancers

3. **Enhancers** (see Enhancer section below)
   - Each enhancer manages a specific functionality
   - Subscribes to relevant stores for state updates
   - Updates DOM when state changes

### 2. State Management (Zustand Stores)

All state is managed through **Zustand stores** with the following pattern:

**Store Structure:**
```typescript
const store = create<State & Actions>()(
  persist(
    subscribeWithSelector((set, get) => ({
      // State
      items: [],

      // Actions
      addItem: async (item) => {
        set(state => ({ items: [...state.items, item] }));
        get().calculateTotals();
      }
    })),
    { name: 'storage-key', storage: sessionStorage }
  )
);
```

**Key Stores:**

1. **cartStore** ([src/stores/cartStore.ts](src/stores/cartStore.ts:1))
   - Manages cart items, totals, coupons
   - Actions: `addItem()`, `removeItem()`, `updateQuantity()`, `calculateTotals()`
   - Persists to sessionStorage
   - Calls API to calculate accurate totals with discounts/shipping

2. **campaignStore** ([src/stores/campaignStore.ts](src/stores/campaignStore.ts:1))
   - Stores campaign data (packages, products, shipping methods)
   - Getters: `getPackage()`, `getVariantsByProductId()`
   - Caches campaign data with TTL

3. **checkoutStore** ([src/stores/checkoutStore.ts](src/stores/checkoutStore.ts:1))
   - Manages checkout form state
   - Handles shipping methods, vouchers
   - Coordinates order submission

4. **orderStore** ([src/stores/orderStore.ts](src/stores/orderStore.ts:1))
   - Stores completed order data
   - Manages upsell journey tracking
   - Actions: `loadOrder()`, `addUpsell()`

5. **configStore** ([src/stores/configStore.ts](src/stores/configStore.ts:1))
   - Global configuration (API key, debug mode, etc.)
   - Loads from meta tags or `window.nextConfig`

6. **profileStore** ([src/stores/profileStore.ts](src/stores/profileStore.ts:1))
   - Manages package profiles/mappings
   - Allows swapping packages for different audiences

7. **attributionStore** ([src/stores/attributionStore.ts](src/stores/attributionStore.ts:1))
   - Tracks UTM parameters, referrers, metadata
   - Included in order submissions for tracking

**Important Store Patterns:**

- **Use `get()` for current state**: `const state = get()`
- **Use `set()` to update state**: `set(state => ({ ...state, newValue }))`
- **Call `calculateTotals()` after cart changes**: Ensures UI stays in sync
- **Stores are singletons**: Access via `useCartStore.getState()`

### 3. How Enhancers Work

Enhancers are **JavaScript classes that add functionality to HTML elements** based on data attributes.

**Base Class Hierarchy:**

```
BaseEnhancer (abstract)
├── BaseActionEnhancer (for buttons/actions)
│   └── AddToCartEnhancer, RemoveItemEnhancer, etc.
├── BaseCartEnhancer (for cart-dependent features)
│   └── PackageSelectorEnhancer, CartToggleEnhancer, etc.
└── Other specialized enhancers
    └── DisplayEnhancers, TimerEnhancer, etc.
```

**Enhancer Lifecycle:**

1. **Construction** - Receives HTML element
2. **initialize()** - Set up event listeners, subscribe to stores
3. **update()** - Called when state changes
4. **destroy()** - Clean up listeners when element removed

**Example Enhancer Pattern:**

```typescript
export class AddToCartEnhancer extends BaseActionEnhancer {
  private packageId?: number;
  private clickHandler?: (event: Event) => void;

  public async initialize(): Promise<void> {
    // 1. Read data attributes
    const packageIdAttr = this.getAttribute('data-next-package-id');
    this.packageId = packageIdAttr ? parseInt(packageIdAttr, 10) : undefined;

    // 2. Set up event handlers
    this.clickHandler = this.handleClick.bind(this);
    this.element.addEventListener('click', this.clickHandler);

    // 3. Subscribe to store changes (if needed)
    this.subscribe(useCartStore, (state) => {
      this.updateButtonState(state);
    });
  }

  private async handleClick(event: Event): Promise<void> {
    event.preventDefault();

    await this.executeAction(
      async () => {
        const cartStore = useCartStore.getState();
        await cartStore.addItem({
          packageId: this.packageId,
          quantity: 1,
          isUpsell: false
        });
      },
      { showLoading: true, disableOnProcess: true }
    );
  }

  public update(_data?: any): void {
    // Update DOM based on new state
  }

  public override destroy(): void {
    if (this.clickHandler) {
      this.element.removeEventListener('click', this.clickHandler);
    }
    super.destroy();
  }
}
```

**Key Enhancer Categories:**

1. **Action Enhancers** - Handle user interactions (clicks, inputs)
   - `AddToCartEnhancer`, `RemoveItemEnhancer`, `CheckoutFormEnhancer`
   - Extend `BaseActionEnhancer` which provides loading states

2. **Display Enhancers** - Show dynamic data from stores
   - `CartDisplayEnhancer`, `ProductDisplayEnhancer`, `OrderDisplayEnhancer`
   - Use template syntax: `data-next-display="cart.total.formatted"`

3. **Selector Enhancers** - Manage package selection UI
   - `PackageSelectorEnhancer` - Handles radio/button selectors for packages
   - Emits `selector:item-selected` events for other enhancers

4. **Conditional Enhancers** - Show/hide elements based on state
   - `ConditionalDisplayEnhancer` - Uses `data-next-show` or `data-next-hide`
   - Supports complex conditions: `cart.total > 50 && !cart.isEmpty`

### 4. Data Attributes Reference

**Common Patterns:**

- `data-next-action="add-to-cart"` - Action buttons
- `data-next-display="cart.total.formatted"` - Display dynamic values
- `data-next-package-id="123"` - Reference a package
- `data-next-selector-id="main-selector"` - Reference a selector
- `data-next-show="cart.hasItems"` - Conditional visibility
- `data-next-checkout` - Checkout form marker

**Display Path Format:**
```
data-next-display="object.property.subproperty"

Objects: cart, campaign, order, product, selection
Examples:
- cart.total.formatted → "$49.99"
- cart.items[0].title → First item name
- order.ref_id → Order reference ID
- product[123].price → Package 123 price
```

## Development Guidelines

### When Working with Stores

1. **Always call `calculateTotals()` after cart modifications**
   ```typescript
   await cartStore.addItem({ packageId: 123, quantity: 1 });
   cartStore.calculateTotals(); // Critical!
   ```

2. **Use `getState()` for one-time reads**
   ```typescript
   const cartStore = useCartStore.getState();
   const items = cartStore.items;
   ```

3. **Use `subscribe()` for reactive updates**
   ```typescript
   useCartStore.subscribe((state) => {
     console.log('Cart updated:', state.total);
   });
   ```

4. **Persist important data**
   - Cart items, checkout state → sessionStorage
   - Attribution, profiles → localStorage + sessionStorage

### When Creating Enhancers

1. **Extend the appropriate base class**
   - `BaseActionEnhancer` for buttons/forms
   - `BaseCartEnhancer` for cart-dependent features
   - `BaseEnhancer` for everything else

2. **Always implement required methods**
   ```typescript
   public async initialize(): Promise<void> { }
   public update(data?: any): void { }
   public override destroy(): void { super.destroy(); }
   ```

3. **Clean up in destroy()**
   - Remove event listeners
   - Unsubscribe from stores (handled by BaseEnhancer if using `this.subscribe()`)

4. **Use helper methods from BaseEnhancer**
   - `this.getAttribute()`, `this.setAttribute()`
   - `this.addClass()`, `this.removeClass()`
   - `this.emit()`, `this.on()` for events

### When Modifying State

1. **Use immutable updates**
   ```typescript
   // ✅ Good
   set(state => ({
     ...state,
     items: [...state.items, newItem]
   }));

   // ❌ Bad
   state.items.push(newItem); // Mutates state
   ```

2. **Batch related updates**
   ```typescript
   // ✅ Good - single update
   set({
     items: newItems,
     total: newTotal,
     isEmpty: newItems.length === 0
   });

   // ❌ Bad - multiple renders
   set({ items: newItems });
   set({ total: newTotal });
   set({ isEmpty: newItems.length === 0 });
   ```

3. **Use async actions for API calls**
   ```typescript
   addItem: async (item) => {
     // Update optimistically
     set(state => ({ items: [...state.items, item] }));

     // Then sync with API
     await get().calculateTotals(); // This calls the API
   }
   ```

### Common Patterns

**Reading a Package from Campaign Store:**
```typescript
const campaignStore = useCampaignStore.getState();
const packageData = campaignStore.getPackage(packageId);

if (!packageData) {
  throw new Error(`Package ${packageId} not found`);
}

const price = parseFloat(packageData.price_total);
const title = packageData.name;
```

**Adding an Item to Cart:**
```typescript
const cartStore = useCartStore.getState();
await cartStore.addItem({
  packageId: 123,
  quantity: 1,
  isUpsell: false
});
```

**Emitting an Event:**
```typescript
import { EventBus } from '@/utils/events';

const eventBus = EventBus.getInstance();
eventBus.emit('cart:item-added', {
  packageId: 123,
  quantity: 1
});
```

**Subscribing to State Changes:**
```typescript
// In an enhancer's initialize() method
this.subscribe(useCartStore, (state) => {
  this.updateDisplay(state);
});
```

## Important Implementation Details

### Cart Totals Calculation

The cart store's `calculateTotals()` method:
1. **Calls the API** to get accurate pricing with discounts/shipping
2. Updates the `totals` object with formatted values
3. Updates enriched items with discount information
4. Should be called after **every cart modification**

```typescript
// This is NOT a simple calculation - it makes an API call!
await cartStore.calculateTotals();
```

### Package vs Product

- **Package** = A sellable SKU with a specific price/quantity
  - Example: "3-pack bottles for $49.99"
  - Has `ref_id`, `price_total`, `qty`, etc.

- **Product** = Base product with variants
  - Example: "Premium Supplement"
  - Can have multiple packages with different quantities/prices

### Profile System

Profiles allow **package ID mapping** for A/B testing or audience segmentation:

```typescript
// Register a profile
profileStore.registerProfile({
  id: 'vip',
  name: 'VIP Customers',
  packageMappings: {
    123: 456  // Map package 123 → 456 for VIP customers
  }
});

// Apply profile
await profileManager.applyProfile('vip');

// Now when user adds package 123, they get 456 instead
```

### Event System

The SDK uses a **centralized event bus** for decoupled communication:

```typescript
import { EventBus } from '@/utils/events';
const eventBus = EventBus.getInstance();

// Emit
eventBus.emit('cart:item-added', { packageId: 123 });

// Listen
eventBus.on('cart:item-added', (data) => {
  console.log('Item added:', data.packageId);
});
```

**Common Events:**
- `cart:item-added`, `cart:item-removed`, `cart:updated`
- `selector:item-selected`, `selector:selection-changed`
- `checkout:submitted`, `order:completed`
- `upsell:added`, `upsell:declined`

## Common Pitfalls to Avoid

1. **Don't mutate state directly**
   ```typescript
   // ❌ Bad
   state.items.push(newItem);

   // ✅ Good
   set(state => ({ items: [...state.items, newItem] }));
   ```

2. **Don't forget to call calculateTotals()**
   ```typescript
   // ❌ Bad - totals won't update
   await cartStore.addItem(item);

   // ✅ Good
   await cartStore.addItem(item);
   await cartStore.calculateTotals();
   ```

3. **Don't create multiple enhancer instances for the same element**
   - The AttributeScanner handles this automatically
   - Each element should only have ONE enhancer instance per type

4. **Don't access stores during module initialization**
   ```typescript
   // ❌ Bad - store might not be ready
   const cartStore = useCartStore.getState();
   export const myFunction = () => { ... };

   // ✅ Good - access inside functions
   export const myFunction = () => {
     const cartStore = useCartStore.getState();
     // ...
   };
   ```

5. **Don't forget to clean up in destroy()**
   ```typescript
   public override destroy(): void {
     // Remove ALL event listeners you added
     this.element.removeEventListener('click', this.handler);

     // Call parent destroy
     super.destroy();
   }
   ```

## Working with This Codebase

When making changes:

1. **Understand the flow**: SDK Init → Store Init → DOM Scan → Enhancer Creation
2. **Follow the patterns**: Use existing enhancers as templates
3. **Test with debug mode**: Always enable debug mode during development
4. **Check store state**: Use `window.nextDebug.stores` to inspect state
5. **Emit appropriate events**: Other enhancers may be listening
6. **Document data attributes**: Update AttributeParser if adding new attributes
7. **Type everything**: Maintain TypeScript strict mode compliance

## Summary

This SDK is built on three pillars:

1. **Stores (Zustand)** - Centralized state management with persistence
2. **Enhancers (Classes)** - Progressive enhancement of HTML elements
3. **Data Attributes** - Declarative configuration in HTML

When you understand how these work together, the entire codebase becomes clear!
